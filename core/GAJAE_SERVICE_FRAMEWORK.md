# 📋 가재 서비스 도입 프레임워크 (Gajae Service Adoption Framework)

본 문서는 가재 컴퍼니가 구축하는 모든 신규 서비스(Service)가 가재 코어(Gajae Core)의 지능과 규율을 어떻게 계승하고 적용해야 하는지 정의합니다.

---

## 1. 지능 계승 원칙 (Intelligence Inheritance)

모든 신규 프로젝트는 가재 코어의 '뿌리 지능'을 기반으로 설계되어야 하며, 다음의 구조를 강제한다.

### 1.1. 코어 규율 강제 적용 (Mandatory Rules)
- **대상**: `core/rules/RULES_CONSTITUTION.md` (헌법), `RULES_OPERATIONS.md` (공정) 등.
- **적용**: 프로젝트 기동 시 `core/rules`의 모든 파일을 참조하거나 링크하여 해당 프로젝트의 '절대 규율'로 삼는다.
- **예외**: 프로젝트 특성에 따른 세부 공정 변경은 허용되나, **헌법 제5조(지능의 독립성)** 및 **제6조(기록의 투명성)**는 어떠한 경우에도 침해할 수 없다.

### 1.2. 페르소나의 일관성 (Persona Consistency)
- 모든 프로젝트에서 가재 군단(PO, DEV, UX 등)은 가재 코어(`core/persona/`)에서 정의된 고유의 성격과 말투를 유지한다.
- 프로젝트의 도메인(예: 금융, 예술, 커머스 등)에 따라 해당 페르소나에 '추가 전문 지식'을 주입할 수 있으나, 그 영혼의 본질은 변하지 않는다.

---

## 2. Build in Public (BIP) 표준 공정

미래의 모든 서비스는 개발 과정 자체가 제품이 되는 BIP 형식을 따라야 한다.

1. **실시간 역사 박제 (Chronicle)**: 
    - 모든 연산은 `chronicle/daily/` 폴더에 날짜별로 기록한다.
    - 지휘관(CEO)의 지시부터 가재들의 고뇌(Thinking)가 누락 없이 보존되어야 한다.
2. **지능의 자산화 (Assetization)**:
    - 단순한 코드 결과물이 아닌, 해당 기능을 구현하기 위해 사용된 **'최종 시스템 프롬프트'**와 **'의사결정 로직'**을 별도의 자산(`assets/`)으로 분류하여 관리한다.
3. **투명한 거버넌스 (Governance)**:
    - HR의 정기 감사 보고서를 공개하여 프로젝트의 건강도를 외부 팬덤에게 증명한다.

---

## 3. 신규 프로젝트 셋업 가이드 (Setup Guide)

1. **레포지토리 생성**: 신규 서비스 전용 레포지토리(예: `gajae-new-service`)를 생성한다.
2. **코어 연결**: 가재 코어(`yuna-openclaw`)를 프로젝트의 'Root Dependency'로 설정한다.
3. **규율 동기화**: `core/rules/`의 공정을 해당 프로젝트의 개발 환경에 이식한다.
4. **기록 시작**: 첫 연산부터 `chronicle/`에 지능의 흔적을 남기며 빌드인퍼블릭을 시작한다.
