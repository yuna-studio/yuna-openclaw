# 🦞 가재 컴퍼니 코딩 헌법 (RULES_CODING.md)

이 규칙은 가재 컴퍼니의 모든 개발 프로젝트에 예외 없이 적용되는 영구 규칙입니다.

## 1. 아키텍처 원칙 (Strict Clean Architecture)
- **Layer Separation:** 의존성 분리와 관심사 분리를 철저히 엄수한다.
- **Passive View:** 뷰는 철저히 수동적(Passive)이어야 한다. 직접 로직을 갖지 않고 데이터만 출력한다.
- **ViewModel (State):** 상태 관리와 뷰 노출용 데이터 가공은 뷰모델에서 전담한다.
- **UseCase (Logic Orchestration):** 데이터를 조합하고 비즈니스 로직을 전달하는 핵심 코어는 유즈케이스가 담당한다.

## 2. 구현 및 코드 퀄리티
- **Null Safety:** `!` (Null Assertion) 연산자 사용을 절대 금지한다.
- **Documentation:** 유지보수를 위해 모든 핵심 로직에 상세한 주석을 작성한다.
- **Modular Logging:** 로그 모델을 추상화하여 모듈화한다. 파이어베이스(또는 Mixpanel) 연동 시 추상화된 모듈을 사용한다.

## 3. 커밋 및 협업 규칙
- **Commit Strategy:** 기능 단위로 잘게 쪼개어 목적에 맞게 커밋한다.
- **Commit Message:** 반드시 **한글**로 작성하며, 커밋의 목적을 명확히 밝힌다.
- **Build Integrity (P0):** DEV가재는 커밋 전 반드시 로컬 빌드 및 린트(Lint) 체크를 수행하여 **'신택스 에러 제로'** 및 **'빌드 성공 상태'**를 확인해야 한다. 깨진 코드를 커밋하는 것은 가재의 수치로 간주한다.
- **Escalation (Build Fail):** 만약 기술적 난제로 인해 빌드 실패가 지속될 경우, 독단적으로 시간을 끌지 말고 즉시 사회자에게 **'프로세스 검토 요청'**을 보고해야 한다. 대표님께 공식적으로 이슈 보고가 된 건에 대해서는 징계가 면제되며, 대표 레벨에서 프로세스를 보완한다.
- **Self-Execution:** 모든 가재는 태스크 완료 시 스스로 커밋과 푸시를 완료하여 형상 관리에 기여한다.
- **Figma-Level Align:** UX가재의 명세를 픽셀 단위로 숙지하고 모호한 점은 질문하여 수치화한다.

## 3. 현미경 설계 및 문서화 룰 (Micro-Documentation)
DEV가재는 구현 전, 다음 항목을 포함한 상세 설계를 수행해야 한다.
1. **요구사항 정합성:** 설계가 비즈니스 요구사항(docs/requirement.md)을 100% 충족하는지 자가 검토.
2. **레이어별 명세:** 각 레이어(Domain, Data, Presentation)에 필요한 핵심 함수명, 파라미터, 반환 타입을 명시.
3. **DB 및 쿼리:** DB 스키마 설계와 함께 이를 생성/조회하기 위한 SQL 쿼리문을 문서에 포함.
4. **승인 프로세스:** 위 내용이 담긴 정밀 설계안에 대해 대표님의 최종 승인을 받은 후 조립 시작.
